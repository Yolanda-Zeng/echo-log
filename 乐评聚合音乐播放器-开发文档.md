# 乐评聚合音乐播放器 — 开发文档

版本：v1.0
项目性质：个人使用的学习型 Web 项目

---

## 1. 项目概述

一个聚合多个乐评网站评分数据、并支持音乐播放的 Web 应用。核心特色是首页用 3D 动效（Kinetic Images 风格）展示 Billboard 100 专辑封面，点击某张封面进入专辑详情页，查看聚合评分与精选短评，并可直接播放。

**定位**：个人学习项目，非商业产品。单用户使用，无需账号系统。

**核心价值**：把分散在多个乐评网站的评分/短评聚合到一处，同时提供即时试听，减少"查评分—开播放器"的来回切换。

 **产品名称**：**回响录 / Echo Log**
 **简介（中）**：每一次聆听都会留下回响，回响录把这些散落的声音重新收拢，变成可以翻阅的乐评档案。
 **Tagline (EN)**: *A log of echoes — where scattered listening becomes a shared archive.*

---

## 2. 范围界定（Scope）

### 2.1 包含的功能
- 首页：Billboard 100 专辑的 3D 动效陈列（Spiral / Tower 形态），点击封面跳转详情
- 专辑详情页：播放功能 + 多源评分对比 + 精选短评（标注来源）

### 2.2 明确不做（当前阶段）
- 用户账号系统 / 登录 / 多设备同步
- 搜索页、个人收藏页、设置页
- RateYourMusic 数据源（反爬难度过高，个人项目性价比低）
- 完整评论列表聚合（只做精选摘要，不做评论区式的完整展示）
- 移动端适配（先做桌面 Web，视情况再扩展）

> 以上"不做"的部分并非永久排除，而是第一阶段有意收窄范围，避免范围蔓延导致项目搁置。

## 3. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 前端框架 | React (Vite) | 轻量，配合 R3F 生态成熟 |
| 3D 渲染 | React Three Fiber + drei | 首页专辑陈列动效 |
| 状态管理 | Zustand | 轻量，够用即可，不上 Redux |
| 后端 | Node.js (Express/Fastify) | 与网易云接口生态（JS 实现较多）配合顺畅 |
| 数据库 | SQLite | 单用户场景，无需上 Postgres |
| 定时任务 | node-cron | 定期抓取乐评数据并缓存，不做实时爬取 |
| 音乐数据源 | 网易云音乐非官方接口（如 NeteaseCloudMusicApi） | 获取搜索、播放链接、歌词 |
| 乐评数据源 | 豆瓣音乐（爬虫）+ AOTY 或 Pitchfork（爬虫） | 定时抓取后入库，前端只读本地缓存 |

## 4. 系统架构

```
┌─────────────┐        ┌──────────────────┐        ┌─────────────────┐
│   前端 (React)│ <────> │  后端 API (Node)   │ <────> │   SQLite 数据库   │
│  R3F 首页动效  │  REST  │  - 专辑聚合接口     │        │  albums          │
│  播放器组件    │        │  - 播放代理接口     │        │  reviews         │
│  评分展示组件  │        │                    │        │  play_cache      │
└─────────────┘        └──────────────────┘        └─────────────────┘
                                  │
                                  │ 定时任务 (node-cron)
                                  ▼
                    ┌───────────────────────────┐
                    │   爬虫模块 (独立脚本/任务)     │
                    │  - 豆瓣音乐抓取器             │
                    │  - AOTY/Pitchfork 抓取器      │
                    │  - 匹配/归一化逻辑             │
                    └───────────────────────────┘
                                  │
                                  ▼
                    ┌───────────────────────────┐
                    │   网易云音乐接口（外部）        │
                    │  搜索 / 播放链接 / 歌词          │
                    └───────────────────────────┘
```

**关键设计原则**：爬虫模块与主应用解耦，独立定时运行，写入数据库后主应用只读缓存数据，避免前端请求触发实时爬取（防封 IP，也保证响应速度）。

## 5. 数据模型

### 5.1 `albums` 表（专辑主表）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | 内部主键 |
| title | TEXT | 专辑名 |
| artist | TEXT | 艺人名 |
| year | INTEGER | 发行年份 |
| cover_url | TEXT | 封面图地址 |
| netease_id | TEXT | 网易云对应 ID（用于播放） |
| billboard_rank | INTEGER | Billboard 100 排名（首页展示用） |
| created_at | DATETIME | 入库时间 |

### 5.2 `reviews` 表（乐评/评分表）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| album_id | INTEGER FK | 关联 albums.id |
| source | TEXT | 数据源：douban / aoty / pitchfork |
| score | REAL | 原始评分 |
| score_normalized | REAL | 归一化到 10 分制后的分数 |
| excerpt | TEXT | 精选短评摘要（1-2 条） |
| source_url | TEXT | 原文链接，供用户跳转查看完整内容 |
| fetched_at | DATETIME | 抓取时间 |

### 5.3 匹配逻辑说明
不同数据源的专辑条目需要关联到同一个 `albums.id`：
- 一期方案：按「艺人名（归一化：去除大小写/空格差异）+ 专辑名（归一化）+ 年份」做规则匹配
- 匹配失败的条目先存入待处理队列，人工看一眼决定是否手动关联（数据量只有 100 张专辑，人工兜底完全可行）
- 不建议一期就上模糊字符串相似度算法（如 Levenshtein），规则匹配 + 人工兜底对 100 条数据量级来说性价比更高

## 6. 页面与功能详细设计

### 6.1 首页 — Billboard 100 动效陈列

**功能点**：
- 加载 Billboard 100 专辑封面，用 R3F 渲染成 Spiral 或 Tower 形态（先选一种实现，另一种作为二期切换选项）
- 自动缓慢旋转 + 支持鼠标拖拽微调视角
- 点击某张封面：
  - 触发相机聚焦动画（放大靠近该封面）
  - 使用 drei 的 `<Html>` 组件叠加一个轻量信息卡（专辑名+综合评分），停留片刻后跳转详情页；或直接跳转，视觉过渡看实现难度取舍
- 加载状态：封面图异步加载时用纯色占位块（可用专辑主色调，参考此前 hover-reveal demo 的取色思路）

**技术要点**：
- 100 个固定位置的螺旋/圆柱坐标计算（参数方程，非动态无限网格，性能压力可控）
- 纹理加载注意做懒加载/分批加载，避免 100 张图同时加载卡顿
- 点击检测用 R3F 的 raycasting（`onClick` 事件已封装好，无需手写射线逻辑）

### 6.2 专辑详情页

**功能点**：
- 专辑基本信息：封面、艺人、年份
- 播放器：接入网易云接口播放该专辑歌曲，含播放/暂停、进度条、歌曲列表
- 评分展示：多源评分并列展示（如豆瓣 8.5 / AOTY 78，归一化后可加一个"综合分"）
- 精选短评：每个来源 1-2 条摘要，标注来源图标/名称，附原文链接
- 返回首页入口（回到 3D 陈列视角）

## 7. 爬虫模块设计

### 7.1 豆瓣音乐抓取器
- 频率控制：单 IP 内合理间隔请求（建议 2-5 秒随机延迟），避免触发封禁
- 登录态：视情况决定是否携带 cookie（携带 cookie 时豆瓣的限制策略是封 cookie 不封 IP，更利于个人项目长期运行）
- 抓取内容：专辑评分、短评列表（取点赞数最高的前几条作为"精选"）

### 7.2 AOTY / Pitchfork 抓取器
- 优先选择反爬相对温和的一个先实现（可在实际开发时用少量请求测试两者的响应情况再定）
- 抓取内容：评分（Critic Score / User Score）、编辑短评摘要

### 7.3 定时任务
- 使用 node-cron，建议频率：每天 1 次全量更新即可（Billboard 100 数据本身更新不频繁，没必要高频抓取）
- 抓取结果写入 `reviews` 表，覆盖旧数据或做版本追加（看是否需要历史评分变化趋势，一期建议直接覆盖，简单够用）

## 8. 开发路线图

**阶段一：播放链路打通**
- 接入网易云接口，实现搜索、播放、歌词基本功能
- 做一个最简单的专辑详情页（无评分数据，先验证播放能跑通）

**阶段二：单一乐评源接入**
- 完成豆瓣抓取器，写入数据库
- 详情页接入豆瓣评分和短评展示

**阶段三：首页 3D 动效**
- 实现 Spiral/Tower 陈列的静态版本（占位图即可）
- 加入点击跳转详情页的交互
- 替换为真实 Billboard 100 数据

**阶段四：第二乐评源 + 匹配逻辑**
- 接入 AOTY 或 Pitchfork
- 实现跨源匹配规则，处理匹配失败的兜底逻辑

**阶段五：打磨**
- 加载状态、错误处理、样式细节
- 视情况决定是否补充二期功能（搜索、收藏等）

> 建议严格按此顺序推进，每个阶段都产出一个能跑通的最小闭环，避免多个模块同时开工导致互相阻塞调试。

## 9. 风险与应对

| 风险 | 应对方案 |
|---|---|
| 乐评网站反爬升级，抓取失败 | 定时任务失败时保留上次缓存数据，不影响前端展示；抓取脚本与主应用解耦，出问题不影响播放功能 |
| 网易云接口不稳定（非官方接口，说停就停） | 播放功能失效时前端应有明确的降级提示，而不是白屏；预留切换到本地文件播放的可能性 |
| 跨源专辑匹配出错（张冠李戴） | 100 张数据量级下人工核对一遍完全可行，一期不必追求全自动完美匹配 |
| 3D 效果开发耗时超预期 | 阶段三设定"先用占位图验证交互"的中间目标，避免真实数据接入前卡在视觉细节上 |
